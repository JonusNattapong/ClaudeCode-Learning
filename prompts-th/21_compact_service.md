# บริการบีบอัด (การสรุปบทสนทนา) (Compact Service)

> **พบใน**: สถาปัตยกรรมภายในของ Claude Code
> **ตัวแปร:** `NO_TOOLS_PREAMBLE`, `BASE_COMPACT_PROMPT`, `PARTIAL_COMPACT_PROMPT`, `PARTIAL_COMPACT_UP_TO_PROMPT`
> **โมเดล:** Sonnet (ผ่านการ fork ที่แชร์แคช หรือ streaming fallback)

---

## วัตถุประสงค์ (Purpose)

สร้างการสรุปบทสนทนาอย่างละเอียดเมื่อหน้าต่างบริบท (context window) เข้าใกล้ขีดจำกัด บริการบีบอัดมีโหมดการทำงานสามโหมด: การบีบอัดแบบเต็มรูปแบบ, การบีบอัดบางส่วนของข้อความล่าสุด และการบีบอัดบางส่วนของข้อความเก่า

---

## คำนำสั่งห้ามใช้เครื่องไม้เครื่องมือ (No-Tools Preamble)

ถูกใส่เข้าไปเป็นคำแนะนำแรกเพื่อป้องกันไม่ให้โมเดลเรียกใช้เครื่องมือในระหว่างการสรุปข้อมูล สิ่งนี้สำคัญมากเพราะการ fork แบบแชร์แคชจะสืบทอดชุดเครื่องมือทั้งหมดมาจากตัวหลัก

```
สำคัญมาก: ให้ตอบกลับเป็นข้อความเท่านั้น (TEXT ONLY) ห้ามเรียกใช้เครื่องมือใดๆ ทั้งสิ้น

- ห้ามใช้ Read, Bash, Grep, Glob, Edit, Write หรือเครื่องมืออื่นๆ "เด็ดขาด"
- คุณมีบริบททั้งหมดที่จำเป็นอยู่ในการสนทนาด้านบนแล้ว
- การเรียกใช้เครื่องมือจะถูกปฏิเสธ (REJECTED) และจะเป็นการเสียเทิร์นเดียวที่คุณมีไปเปล่าๆ — คุณจะล้มเหลวในงานนี้
- การตอบกลับทั้งหมดของคุณต้องเป็นข้อความปกติ: เริ่มด้วยบล็อก <analysis> ตามด้วยบล็อก <summary>
```

## พรอมต์บีบอัดพื้นฐาน (การสรุปแบบเต็มรูปแบบ) (Base Compact Prompt)

ใช้เมื่อมีการสรุปบทสนทนาทั้งหมด:

```
งานของคุณคือสร้างสรุปรายละเอียดของบทสนทนาที่ผ่านมาทั้งหมด โดยให้ความสำคัญอย่างใกล้ชิดกับคำขอที่ชัดเจนของผู้ใช้และการดำเนินการก่อนหน้าของคุณ
สรุปนี้ควรมีความละเอียดในการรวบรวมรายละเอียดทางเทคนิค, รูปแบบโค้ด (code patterns) และการตัดสินใจเชิงสถาปัตยกรรมที่จำเป็นสำหรับการทำงานพัฒนาต่อไปโดยไม่สูญเสียบริบท
```

### ส่วนของสรุปที่บังคับ (Required Summary Sections)

1. **คำขอและเจตนาหลัก (Primary Request and Intent)** — คำขอและเจตนาที่ชัดเจนทั้งหมดของผู้ใช้
2. **แนวคิดทางเทคนิคที่สำคัญ (Key Technical Concepts)** — เทคโนโลยี, เฟรมเวิร์ก, รูปแบบที่ได้พูดคุยกัน
3. **ไฟล์และส่วนของโค้ด (Files and Code Sections)** — ไฟล์ที่ตรวจสอบ/แก้ไข/สร้าง พร้อมโค้ดบางส่วนที่สมบูรณ์
4. **ข้อผิดพลาดและการแก้ไข (Errors and Fixes)** — ข้อผิดพลาดที่พบและขั้นตอนการแก้ไข
5. **การแก้ปัญหา (Problem Solving)** — ปัญหาที่ได้รับการแก้ไขแล้วและการแก้ไขปัญหาที่กำลังดำเนินอยู่
6. **ข้อความทั้งหมดจากผู้ใช้ (All User Messages)** — ทุกข้อความจากผู้ใช้ที่ไม่ใช่ผลลัพธ์ของเครื่องมือ (สำคัญมากในการติดตามการเปลี่ยนทิศทางของเจตนา)
7. **งานที่ค้างอยู่ (Pending Tasks)** — รายการงานที่ยังทำไม่เสร็จ
8. **งานปัจจุบัน (Current Work)** — คำอธิบายที่แม่นยำของงานที่กำลังทำอยู่ก่อนการสรุปข้อมูล
9. **ขั้นตอนถัดไป (ทางเลือก) (Optional Next Step)** — ขั้นตอนถัดไปเฉพาะในกรณีที่สอดคล้องโดยตรงกับคำขอเล่าสุดของผู้ใช้

---

## พรอมต์บีบอัดบางส่วน (ข้อความล่าสุด) (Partial Compact Prompt)

ใช้เมื่อมีการสรุปเฉพาะข้อความล่าสุดในขณะที่ยังคงบริบทเก่าไว้:

```
งานของคุณคือสร้างสรุปรายละเอียดของส่วน "ล่าสุด" ของบทสนทนา — ข้อความที่ต่อมาจากบริบทก่อนหน้าที่ถูกเก็บไว้ ข้อความก่อนหน้านั้นจะถูกเก็บรักษาไว้ตามเดิมและ "ไม่" จำเป็นต้องสรุป ให้เน้นการสรุปของคุณไปที่สิ่งที่ได้พูดคุย, เรียนรู้ และทำสำเร็จในข้อความล่าสุดเท่านั้น
```

---

## พรอมต์บีบอัดบางส่วนถึงจุดที่กำหนด (บริบทเก่า) (Partial Compact Up-To Prompt)

ใช้เมื่อสรุปข้อความเก่าเพื่อเพิ่มพื้นที่ โดยยังคงรักษาข้อความใหม่ไว้:

```
งานของคุณคือสร้างสรุปรายละเอียดของบทสนทนานี้ สรุปนี้จะถูกวางไว้ที่จุดเริ่มต้นของเซสชันที่ดำเนินต่อ; ข้อความใหม่กว่าที่สร้างขึ้นจากบริบทนี้จะตามมาหลังจากสรุปของคุณ (คุณจะไม่เห็นพวกมันที่นี่) ให้สรุปอย่างละเอียดเพื่อให้คนที่อ่านเพียงสรุปของคุณและข้อความใหม่สามารถเข้าใจสิ่งที่เกิดขึ้นและทำงานต่อไปได้อย่างสมบูรณ์
```

จะมีการเปลี่ยนส่วนที่ 8 ("งานปัจจุบัน") เป็น "งานที่ทำเสร็จแล้ว" และส่วนที่ 9 เป็น "บริบทสำหรับการทำงานต่อไป"

## บล็อกการวิเคราะห์ (Analysis Block)

ทั้งสามรูปแบบข้างต้นจะรวม `DETAILED_ANALYSIS_INSTRUCTION` ที่กำหนดให้โมเดลห่อหุ้มความคิดไว้ในแท็ก `<analysis>` ก่อนจะเขียน `<summary>` บล็อกการวิเคราะห์นี้เปรียบเสมือนกระดาษร่างที่จะถูกตัดออกโดยฟังก์ชัน `formatCompactSummary()` ก่อนที่สรุปจะถูกส่งเข้าสู่บริบทจริง

```
ก่อนจะเขียนสรุปสุดท้าย ให้ห่อหุ้มการวิเคราะห์ของคุณไว้ในแท็ก <analysis> เพื่อจัดระเบียบความคิดและตรวจสอบให้แน่ใจว่าคุณได้ครอบคลุมประเด็นที่จำเป็นทั้งหมดแล้ว ในกระบวนการวิเคราะห์ของคุณ:

1. วิเคราะห์แต่ละข้อความและส่วนของบทสนทนาตามลำดับเวลา สำหรับแต่ละส่วนให้ระบุอย่างละเอียดถึง:
   - คำขอและเจตนาที่ชัดเจนของผู้ใช้
   - แนวทางของคุณในการจัดการคำขอของผู้ใช้
   - การตัดสินใจสำคัญ, แนวคิดทางเทคนิค และรูปแบบโค้ด
   - รายละเอียดเฉพาะ เช่น: ชื่อไฟล์, โค้ดบางส่วนที่สมบูรณ์, ลายเซ็นฟังก์ชัน, การแก้ไขไฟล์
   - ข้อผิดพลาดที่คุณพบและวิธีที่คุณแก้ไข
   - ให้ความสำคัญเป็นพิเศษกับข้อเสนอแนะเฉพาะจากผู้ใช้
2. ตรวจสอบความถูกต้องทางเทคนิคและความสมบูรณ์อีกรอบ
```

## การสนับสนุนคำแนะนำที่กำหนดเอง (Custom Instructions Support)

พรอมต์บีบอัดรองรับคำแนะนำการสรุปที่ผู้ใช้จัดเตรียมไว้ให้ผ่าน CLAUDE.md หรือ hooks:

```
อาจมีคำแนะนำการสรุปเพิ่มเติมจัดเตรียมไว้ให้ในบริบทที่รวมอยู่ด้วย
หากมี อย่าลืมปฏิบัติตามคำแนะนำเหล่านี้เมื่อสร้างสรุปข้อมูลข้างต้น
```
