# สกิลการทำความสะอาดโค้ด (Simplify Skill)

> **ประเภท**: สกิลที่แถมมาในตัว (Bundled skill - ถูกคอมไพล์เข้าไปในไฟล์ไบนารีของ Claude Code)
>
> **การเรียกใช้งาน**: `/simplify` หรือ `/simplify focus on [พื้นที่ที่ต้องการเน้น]`
>
> คำสั่ง `/simplify` เป็นสกิลในตัวที่รันการรีวิวโค้ดแบบหลายเอเจนต์โดยอัตโนมัติและทำการทำความสะอาดไฟล์ที่เพิ่งเปลี่ยนแปลงล่าสุด ถูกออกแบบมาให้รันหลังจากนำฟีเจอร์ไปใช้งานหรือแก้ไขบั๊กเสร็จสิ้น ก่อนที่จะเปิด pull request

---

## ความพร้อมใช้งาน (Availability)

นี่คือ **สกิลที่แถมมาในตัว** หมายความว่ามันถูกส่งมาพร้อมกับไฟล์ไบนารีของ Claude Code และไม่ต้องการการติดตั้ง ต่างจากสกิลที่ผู้ใช้กำหนดเองใน `.claude/skills/` สกิลนี้ได้รับการดูแลโดย Anthropic และอัปเดตพร้อมกับรุ่นที่ออกของ Claude Code ซอร์สโค้ด `SKILL.md` ที่แน่นอนจะถูกคอมไพล์เข้าไปในไบนารีและไม่สามารถเข้าถึงได้โดยสาธารณะ

เนื้อหาด้านล่างนี้ถูกสร้างขึ้นใหม่จากเอกสารสาธารณะ, ข้อมูลจากชุมชน และผลจากการวิเคราะห์พฤติกรรม

---

## ขั้นตอนการทำงาน (Workflow)

### ระยะที่ 1: การตรวจจับการเปลี่ยนแปลง (Change Detection)

สกิลจะระบุขอบเขตการรีวิวโดยตรวจสอบการเปลี่ยนแปลงล่าสุด:

- หากอยู่ใน Git repository จะรัน `git diff` (หรือ `git diff HEAD` สำหรับการเปลี่ยนแปลงที่ staged ไว้) เพื่อระบุสิ่งที่ถูกแก้ไข
- หากไม่พบ Git repository จะรีวิวไฟล์ที่ถูกแก้ไขล่าสุด
- เอาต์พุตของ diff จะเป็นตัวกำหนดขอบเขตที่แน่นอนของการรีวิว

### ระยะที่ 2: การรีวิวแบบขนานด้วยสามเอเจนต์ (Three-Agent Parallel Review)

เอเจนต์ย่อยเฉพาะทางสามตัวจะถูกรันพร้อมกันแบบขนาน โดยแต่ละตัวจะได้รับ diff ฉบับเต็มเพื่อใช้เป็นบริบท:

#### เอเจนต์การนำโค้ดกลับมาใช้ใหม่ (Code Reuse Agent)

รีวิวการเปลี่ยนแปลงเพื่อระบุ:
- ตรรกะที่ซ้ำซ้อนซึ่งสามารถดึงออกมาเป็นฟังก์ชันที่ใช้ร่วมกันได้
- ฟังก์ชันยูทิลิตี้หรือ helper ที่มีอยู่แล้วซึ่งควรนำมาใช้แทนโค้ดใหม่
- บล็อกโค้ดที่ฟุ่มเฟือยและรูปแบบที่ทำซ้ำๆ
- โอกาสในการทำ refactoring ให้เป็นคอมโพเนนต์หรือโมดูลที่ใช้ซ้ำได้

#### เอเจนต์คุณภาพโค้ด (Code Quality Agent)

ประเมินโครงสร้างและสไตล์ของโค้ด:
- ความสอดคล้องของการตั้งชื่อและความสามารถในการอ่าน
- การแยกฟังก์ชันออกเป็นส่วนๆ และความชัดเจนของลอจิกการควบคุม (control flow)
- การปฏิบัติตามมาตรฐานการเขียนโค้ดที่กำหนดไว้ใน `CLAUDE.md`
- "Code smells" เช่น abstraction ที่รั่วไหล (leaky abstractions), โค้ดที่ใช้ข้อความเป็นตัวกำหนดประเภท (stringly-typed code), การซ้อนทับที่ไม่จำเป็น
- การออกแบบที่ซับซ้อนเกินไป (over-engineering), abstraction ที่ไม่จำเป็น หรือการปรับแต่งก่อนเวลาอันควร ("gold-plating")

#### เอเจนต์ประสิทธิภาพ (Efficiency Agent)

วิเคราะห์ประสิทธิภาพและการใช้ทรัพยากร:
- การจัดสรรหน่วยความจำที่ไม่จำเป็นและการคำนวณที่ฟุ่มเฟือย
- ลูป (loops) ที่ได้รับประโยชน์จากการประมวลผลแบบแบตช์หรือการเพิ่มประสิทธิภาพ
- รูปแบบการสอบถามข้อมูลแบบ N+1 และการเข้าถึงไฟล์หรือเครือข่ายที่ไม่มีประสิทธิภาพ
- โอกาสที่พลาดไปในการประมวลผลแบบขนาน (concurrency)
- การเรนเดอร์ซ้ำที่ไม่จำเป็น (ในโค้ด frontend)

### ระยะที่ 3: การรวบรวมข้อมูลและการประยุกต์ใช้การแก้ไข (Aggregation and Fix Application)

เมื่อเอเจนต์ทั้งสามรันการรีวิวเสร็จสิ้น:

1. สิ่งที่พบจะถูกรวบรวมและกำจัดรายการที่ซ้ำกันออก
2. รายการที่ระบุผิดพลาด (false positives) หรือสิ่งที่พบซึ่งไม่คุ้มค่ากับการจัดการจะถูกข้ามไป
3. การแก้ไขจะถูกนำไปใช้โดยตรงกับ codebase
4. รายงานสรุปสิ่งที่เปลี่ยนแปลงจะถูกแสดงให้ผู้ใช้ทราบ

## การใช้งาน (Usage)

```
/simplify                              # รีวิวการเปลี่ยนแปลงล่าสุดทั้งหมด
/simplify focus on memory efficiency   # เน้นเรื่องประสิทธิภาพหน่วยความจำ
/simplify focus on performance         # เจาะจงไปที่ปัญหาประสิทธิภาพ
```

## การทำงานร่วมกับ CLAUDE.md

ประสิทธิภาพของ `/simplify` ขึ้นอยู่กับไฟล์ `CLAUDE.md` ของโปรเจกต์เป็นอย่างมาก เอเจนต์คุณภาพจะใช้มาตรฐานการเขียนโค้ดและความชอบที่กำหนดไว้ที่นั่นเพื่อให้ข้อเสนอแนะสอดคล้องกับข้อตกลงของทีม โปรเจกต์ที่มีไฟล์ `CLAUDE.md` ที่กำหนดไว้ดีจะได้รับผลลัพธ์ที่ดีขึ้นอย่างเห็นได้ชัด

## ลักษณะการออกแบบ (Design Characteristics)

| คุณสมบัติ | รายละเอียด |
|----------|--------|
| **จำนวนเอเจนต์** | เอเจนต์ย่อย 3 ตัว รันขนานกัน |
| **ขอบเขต** | การเปลี่ยนแปลงล่าสุดเท่านั้น (ผ่าน `git diff`) |
| **การประมวลผล** | เอเจนต์รันพร้อมกันเพื่อความรวดเร็ว |
| **โหมดการแก้ไข** | ประยุกต์ใช้โดยอัตโนมัติพร้อมสรุปข้อมูล |
| **False positives** | จะถูกข้ามไปแทนที่จะโต้เถียง |
| **การปรับแต่ง** | ยอมรับพื้นที่ที่ต้องการเน้นเป็นอาร์กิวเมนต์ |
| **การขึ้นต่อกัน** | อ่านไฟล์ `CLAUDE.md` สำหรับมาตรฐานเฉพาะโปรเจกต์ |

## ตำแหน่งในขั้นตอนการทำงาน

ขอแนะนำขั้นตอนการทำงานดังนี้:

```
สร้างฟีเจอร์ → รันการทดสอบ → /simplify → รีวิว diffs → Commit
```

นี่คือตำแหน่งของ `/simplify` ในฐานะเครื่องมือทำความสะอาดหลังการนำไปใช้งานเพื่อปรับโค้ดที่สร้างโดย AI ให้กระชับขึ้น (ซึ่งมักจะเยิ่นเย้อ) มันไม่ได้มาแทนที่การรีวิวโค้ดโดยมนุษย์ แต่ช่วยทำความสะอาดงานส่วนที่เป็นเชิงกลไกให้โดยอัตโนมัติ
